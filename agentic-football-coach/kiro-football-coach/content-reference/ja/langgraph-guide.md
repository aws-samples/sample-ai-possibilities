# LangChainエージェントガイド

LangChainを使用したフットボールエージェント構築の包括的リファレンス。セットアップ、ReActパターン、デプロイメント、強化戦略をカバーします。

## エージェントアーキテクチャ概要

各チームは5人のプレイヤーを持ち、各プレイヤーがエージェントです。エージェントは毎ティック（60回/秒）ゲーム状態を受信し、500ms以内にアクションを返す必要があります。

```
ゲーム状態 (JSON) → エージェント → プレイヤーアクション (JSON)
```

## ReActパターン

LangGraphは`create_react_agent`を介してReAct（Reasoning and Acting）パターンを使用します。エージェントは体系的に：

1. 現在のゲーム状態について**推論**する
2. ツールを呼び出して**行動**する（例：距離計算、シュート角度評価）
3. ツール結果を**観察**する
4. 構造化されたアクションで**応答**する

### コアコンセプト

**`create_react_agent`** — LangGraphでReActエージェントを構築するための主要関数。以下を行うエージェントを作成：
- 現在のゲーム状態を含むプロンプトを受信
- 利用可能なアクションについて推論
- 分析が必要な場合にツールを呼び出し
- 最終的なアクション判断を返す

**ツール統合** — 推論中にエージェントが呼び出せるPython関数をツールとして定義：
- 距離計算
- シュート角度分析
- パスオプション評価
- スタミナ管理チェック

**状態管理** — LangGraphは状態を明示的に管理します。各呼び出し時にエージェントエグゼキューターを通じて会話履歴とカスタム状態（スコアライン、戦術モード、フォーメーション）を渡します。これはStrandsとは異なり、Strandsでは組み込みのエージェント構造で状態が管理されます。

### Strandsとの主な違い

| 側面 | LangGraph | Strands |
|---|---|---|
| エージェント作成 | `create_react_agent()` | `Agent()` |
| 抽象化レベル | 高い — マネージドReActスキャフォールディング | 低い — エージェントループの明示的制御 |
| 状態管理 | 明示的 — チェーン/エグゼキューターを通じて渡す | 組み込み — エージェントメモリ構造 |
| ツール定義 | LangChainツールデコレーター | `@tool`デコレーター |
| 最適な用途 | 構造化されたマルチステップ推論 | カスタムワークフロー、きめ細かいロジック |

## エージェントの構築

### ステップ1：ローカルテスト

ローカル開発サーバーを起動：
```bash
agentcore dev
```

別のターミナルでテスト呼び出しを送信：
```bash
agentcore invoke --dev '{"prompt": "{\"gameState\": {\"ball\": {\"position\": {\"x\": 30, \"y\": 10}}, \"players\": [], \"score\": {\"home\": 0, \"away\": 0}, \"gameTime\": 60}, \"teamId\": 0}"}'
```

### ステップ2：デプロイ

デプロイスクリプトを実行：
```bash
./deploy-local.sh
```

スクリプトが処理する内容：IAMロール作成、ARM64用依存関係パッケージング、S3アップロード、エージェントデプロイ。

**重要な出力：** Agent ARN形式 `arn:aws:bedrock-agentcore:<region>:<account-id>:runtime/<agent-id>` — ポータル登録用に保存。

## デプロイメント

### AgentCoreデプロイメント

`./deploy-local.sh`でデプロイ後：

1. デプロイ確認：`agentcore status`
2. ログ確認：`aws logs tail /aws/bedrock-agentcore/runtimes/<id>-DEFAULT --follow`
3. 変更後の再デプロイ：`./deploy-local.sh`（ARNは同じまま）
4. オプションでプレイヤーポジションごとに1つのエージェントを作成（最大5つのARN）

## 強化の提案

### 1. ガードレールの統合

Bedrock Guardrailsを使用して責任あるAIを実装し、無効または安全でないエージェント出力をフィルタリングします。LangChainのBedrock Guardrails統合ドキュメントを参照してください。

### 2. プロンプトの改善

エージェントの推論をより良くガイドするためにシステムプロンプトを改善：
- エージェントの役割とポジションを明示的に定義
- 戦術哲学をエンコード（ポゼッションベース、カウンターアタック）
- 意思決定の優先順位とリスク許容度を設定
- 行動制約を指定

**プロンプトエンジニアリングのヒント：**
- 優先順位を明確に（リードを守る vs 追いかける）
- アクション説明の曖昧さを減らす
- マッチコンテキストを簡潔に提供 — 現在の判断に必要なもののみ
- リプレイとトレースから観察された行動に基づいて反復
- エッジケースをテスト：数的不利、コーナー守備、時間稼ぎ

### 3. テストの強化

包括的なテストシナリオを実装：
- 危険な位置でポゼッションを失った時のエージェント行動をテスト
- 異常なゲーム状態をテスト（数的不利、コーナー守備、負傷）
- LLMを使用して多様なテストケースを生成しエッジケースをカバー
- LLMの非決定性を活用するため複数の評価を提出（最良の結果が保持）

### 4. パフォーマンスチューニング

- **並列処理** — 並列ツール呼び出しを実装してレイテンシを削減
- **効率的な使用** — 高コストな操作は必要な時のみ実行
- **モデル選択** — タスクの複雑さに基づいて異なるモデルサイズを試す（速度のNova Micro、性能のClaude）
- **コンテキスト管理** — プロンプトを軽量に保つ；無関係な状態で過負荷にしない

### 5. マルチエージェントアーキテクチャ

代替設計を検討：
- プレイヤーポジションごとに別々のエージェント
- スペシャリストエージェント（守備アナリスト、攻撃ストラテジスト）をコーチエージェントが調整
- レイテンシ削減のための外部ツールとしての専用計算関数

## ベストプラクティス

### システムプロンプト

- 役割、ポジション、戦術哲学を定義
- 異なるゲーム状況（勝っている、負けている、引き分け）の優先順位を設定
- 制約をエンコード（ゴールを無防備にしない、必ずトラックバック）

### 状態管理

- ティックごとに関連する状態のみを渡す（完全なマッチ履歴ではない）
- 可能な場合はティック間で相手の傾向を追跡
- ゲーム内状態を軽量に保ち、現在の判断コンテキストに集中

### エージェントループの認識

注意すべき一般的な問題：
- **無限ループ** — エージェントがアクションにコミットせず再評価を繰り返す；反復制限を設定
- **ツール呼び出し失敗** — ツールがエッジケースを処理し、整形されたレスポンスを返すことを確認
- **コンテキストウィンドウの肥大化** — 現在の判断に必要なもののみを渡す
- **早期終了** — 完全な評価を確保するために停止条件を確認
- **矛盾するツール出力** — プロンプトに競合解決ロジックをエンコード

### プロンプトエンジニアリング

- 望ましい行動を具体的に指定
- スコープを指定（どのプレイヤーに指示が適用されるか）
- 1つのメッセージで矛盾する指示を避ける
- 観察されたマッチ行動に基づいて反復
- エッジケースを意図的にテスト
