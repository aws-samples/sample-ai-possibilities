# LangChain 에이전트 가이드

LangChain으로 풋볼 에이전트를 구축하기 위한 종합 참조 자료. 설정, ReAct 패턴, 배포, 강화 전략을 다룹니다.

## 에이전트 아키텍처 개요

각 팀은 5명의 선수를 보유하며, 각 선수가 에이전트입니다. 에이전트는 매 틱(초당 60회)마다 게임 상태를 수신하고 500ms 이내에 액션을 반환해야 합니다.

```
게임 상태 (JSON) → 에이전트 → 선수 액션 (JSON)
```


## ReAct 패턴

LangGraph는 `create_react_agent`를 통해 ReAct (Reasoning and Acting) 패턴을 사용합니다. 에이전트는 체계적으로:

1. 현재 게임 상태에 대해 **추론**
2. 도구를 호출하여 **행동** (예: 거리 계산, 슈팅 각도 평가)
3. 도구 결과를 **관찰**
4. 구조화된 액션으로 **응답**

### 핵심 개념

**`create_react_agent`** — LangGraph에서 ReAct 에이전트를 구축하기 위한 주요 함수. 다음을 수행하는 에이전트를 생성합니다:
- 현재 게임 상태가 포함된 프롬프트 수신
- 사용 가능한 액션에 대해 추론
- 분석이 필요할 때 도구 호출
- 최종 액션 결정 반환

**도구 통합** — 에이전트가 추론 중 호출할 수 있는 Python 함수를 도구로 정의:
- 거리 계산
- 슈팅 각도 분석
- 패스 옵션 평가
- 스태미나 관리 확인

**상태 관리** — LangGraph는 상태를 명시적으로 관리합니다. 각 호출 시 대화 기록과 커스텀 상태(스코어라인, 전술 모드, 포메이션)를 에이전트 실행기를 통해 전달합니다. 이는 내장 에이전트 구조를 통해 상태를 관리하는 Strands와 다릅니다.

### Strands와의 주요 차이점

| 측면 | LangGraph | Strands |
|---|---|---|
| 에이전트 생성 | `create_react_agent()` | `Agent()` |
| 추상화 수준 | 높음 — 관리형 ReAct 스캐폴딩 | 낮음 — 에이전트 루프의 명시적 제어 |
| 상태 관리 | 명시적 — 체인/실행기를 통해 전달 | 내장 — 에이전트 메모리 구조 |
| 도구 정의 | LangChain 도구 데코레이터 | `@tool` 데코레이터 |
| 최적 용도 | 구조화된 다단계 추론 | 커스텀 워크플로, 세밀한 로직 |

## 에이전트 구축

### 1단계: 로컬 테스트

로컬 개발 서버 시작:
```bash
agentcore dev
```

별도 터미널에서 테스트 호출 전송:
```bash
agentcore invoke --dev '{"prompt": "{\"gameState\": {\"ball\": {\"position\": {\"x\": 30, \"y\": 10}}, \"players\": [], \"score\": {\"home\": 0, \"away\": 0}, \"gameTime\": 60}, \"teamId\": 0}"}'
```

### 2단계: 배포

배포 스크립트 실행:
```bash
./deploy-local.sh
```

스크립트가 처리하는 내용: IAM 역할 생성, ARM64용 의존성 패키징, S3 업로드, 에이전트 배포.

**중요한 출력:** Agent ARN 형식 `arn:aws:bedrock-agentcore:<region>:<account-id>:runtime/<agent-id>` — 포털 등록용으로 저장.

## 배포

### AgentCore 배포

`./deploy-local.sh`로 배포 후:

1. 배포 확인: `agentcore status`
2. 로그 확인: `aws logs tail /aws/bedrock-agentcore/runtimes/<id>-DEFAULT --follow`
3. 변경 후 재배포: `./deploy-local.sh` (ARN은 동일하게 유지)
4. 선택적으로 플레이어 포지션별 에이전트 생성 (최대 5개 ARN)

## 강화 제안

### 1. 가드레일 통합

Bedrock Guardrails를 사용하여 잘못되거나 안전하지 않은 에이전트 출력을 필터링하는 책임 있는 AI를 구현합니다. LangChain의 Bedrock 가드레일 통합 문서를 참조하세요.

### 2. 프롬프트 개선

에이전트 추론을 더 잘 안내하도록 시스템 프롬프트를 개선합니다:
- 에이전트의 역할과 포지션을 명시적으로 정의
- 전술 철학 인코딩 (점유율 기반, 역습)
- 의사 결정 우선순위와 위험 허용도 설정
- 행동 제약 조건 지정

**프롬프트 엔지니어링 팁:**
- 우선순위에 대해 명시적으로 (리드 수비 vs 추격전)
- 액션 설명의 모호성 줄이기
- 매치 컨텍스트를 간결하게 제공 — 현재 결정에 필요한 것만
- 리플레이와 트레이스에서 관찰된 행동을 기반으로 반복
- 엣지 케이스 테스트: 수적 열세, 코너 수비, 시간 끌기

### 3. 테스트 강화

포괄적인 테스트 시나리오 구현:
- 위험한 위치에서 점유를 잃었을 때의 에이전트 행동 테스트
- 비정상적인 게임 상태 테스트 (수적 열세, 코너 수비, 부상)
- LLM을 사용하여 엣지 케이스를 포함하는 다양한 테스트 케이스 생성
- LLM의 비결정성을 활용하기 위해 여러 번 평가 제출 (최고 결과 보존)

### 4. 성능 튜닝

- **병렬 처리** — 레이턴시를 줄이기 위해 병렬 도구 호출 구현
- **효율적 사용** — 비용이 큰 작업은 필요할 때만 실행
- **모델 선택** — 작업 복잡도에 따라 다른 모델 크기 시도 (속도를 위한 Nova Micro, 성능을 위한 Claude)
- **컨텍스트 관리** — 프롬프트를 간결하게 유지; 관련 없는 상태로 과부하 방지

### 5. 멀티 에이전트 아키텍처

대안적 설계 고려:
- 다른 선수 포지션별 별도 에이전트
- 코치 에이전트가 조율하는 전문 에이전트 (수비 분석가, 공격 전략가)
- 레이턴시를 줄이기 위한 외부 도구로서의 전용 계산 함수

## 모범 사례

### 시스템 프롬프트

- 역할, 포지션, 전술 철학 정의
- 다양한 게임 상황에 대한 우선순위 설정 (이기고 있을 때, 지고 있을 때, 비기고 있을 때)
- 제약 조건 인코딩 (골문을 비우지 않기, 항상 수비 복귀)

### 상태 관리

- 틱당 관련 상태만 전달 (전체 매치 기록 아님)
- 가능한 경우 틱 간 상대 경향 추적
- 인게임 상태를 간결하게 유지하고 현재 결정 컨텍스트에 집중

### 에이전트 루프 인식

주의해야 할 일반적인 문제:
- **무한 루프** — 에이전트가 액션을 결정하지 않고 재평가; 반복 제한 설정
- **도구 호출 실패** — 도구가 엣지 케이스를 처리하고 올바른 형식의 응답을 반환하는지 확인
- **컨텍스트 윈도우 비대화** — 현재 결정에 필요한 것만 전달
- **조기 종료** — 전체 평가를 보장하기 위해 중지 조건 검토
- **상충하는 도구 출력** — 프롬프트에 충돌 해결 로직 인코딩

### 프롬프트 엔지니어링

- 원하는 행동에 대해 구체적으로
- 범위 지정 (지시가 적용되는 선수)
- 단일 메시지에서 상충되는 지시 피하기
- 관찰된 매치 행동을 기반으로 반복
- 의도적으로 엣지 케이스 테스트
